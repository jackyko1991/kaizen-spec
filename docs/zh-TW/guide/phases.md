# 五個階段

kaizen-spec 依序強制執行五個階段。任何階段都不可跳過。每個階段都是下一個階段的門閘。

```mermaid
flowchart LR
  S1["📋 階段一\n規格對齊"]
  S2["🔴 階段二\n測試紅燈"]
  S3["⚙️ 階段三\n實作"]
  S4["✅ 階段四\n驗收"]
  S5["📖 階段五\n文件"]

  S1 -->|"spec.md\n已提交"| S2
  S2 -->|"所有測試\n均失敗"| S3
  S3 -->|"所有測試\n均通過"| S4
  S4 -->|"驗收\n已記錄"| S5

  style S1 fill:#1f2d45,stroke:#388bfd,color:#cdd9e5
  style S2 fill:#3d1f1f,stroke:#f85149,color:#cdd9e5
  style S3 fill:#1f2d1f,stroke:#56d364,color:#cdd9e5
  style S4 fill:#1f2d1f,stroke:#56d364,color:#cdd9e5
  style S5 fill:#2d2520,stroke:#e3b341,color:#cdd9e5
```

---

## 第 1 階段 — 規格對齊

**發生了什麼：** 技能向你提出一系列結構化問題——每次一個——關於你想建構什麼、輸出應該是什麼、範圍內有什麼、明確排除在外的是什麼，以及存在哪些風險。

**寫入了什麼：** `.kaizen/spec.md`——一份已承諾、已協定的規格。

**門閘：** 在 `.kaizen/spec.md` 提交至 git 之前，第 2 階段的任何工作都不會開始。

**原因：** 沒有規格就開始撰寫程式碼的 agents，有一半的時間在解決錯誤的問題。這個階段耗費 2–5 分鐘，卻能節省數小時的返工。

---

## 第 2 階段 — 測試策略

**發生了什麼：** 技能偵測你的專案技術堆疊（TypeScript、Python、Go 等）並推薦一個測試框架。你確認或選擇替代方案。接著，一個測試撰寫 agent 為規格中的每個驗收標準撰寫測試——並在繼續之前驗證所有測試都**失敗**。

**寫入了什麼：**
- `.kaizen/test-strategy.md`——框架、安裝指令、測試檔案路徑，以及每個測試的清單
- `tests/`——實際的測試檔案（全部失敗）

**門閘：** 如果此階段有任何測試通過，技能會停止並要求你調查——在實作前通過的測試意味著測試有誤，或功能已存在。

**原因：** 在程式碼之後撰寫的測試傾向於測試程式碼的實際寫法，而非已協定的規格。先撰寫測試能強制進行誠實的覆蓋率評估。

---

## 第 3 階段 — 實作

**發生了什麼：** 技能將規格分解為獨立任務，寫入 `.kaizen/tasks.json`，並並行生成子 agents——每個任務一個，各在一個 git 工作樹中。Agents 持續實作直到測試通過，然後將自己標記為完成。

**寫入了什麼：**
- `.kaizen/tasks.json`——任務列表，含狀態、agent 指派與時間戳
- `.kaizen/board.html`——即時看板（在瀏覽器中開啟觀看）
- `.kaizen/kaizen.log`——每次狀態轉換一筆 syslog 條目
- 實作程式碼（在 git 工作樹中，完成後合併）

**WIP 限制：** 「進行中」的任務最多同時 3 個。「審查中」最多 2 個。超過限制會使欄位標題變紅並阻止移動。

**安燈繩：** 若 agent 卡住，它會在看板卡片上設定阻塞標誌，並在 `kaizen.log` 中記錄一筆 `WARN` 條目。你會收到通知並可解除阻塞。

**門閘：** 第 4 階段開始前，所有測試必須通過。

---

## 第 4 階段 — 驗收

**發生了什麼：** 技能執行完整的測試套件。若有任何失敗，不宣告完成——返回第 3 階段修復失敗。若需要手動驗證（例如視覺 UI 檢查），會要求你確認特定項目。

**寫入了什麼：**
- `.kaizen/kaizen.log`——驗收結果與總週期時間

**門閘：** 所有測試通過 + 任何必要的手動確認。

**原因：** 「完成」意味著可運作的軟體，而非「在我的機器上可運作」。

---

## 第 5 階段 — 文件（與第 3 階段並行）

**發生了什麼：** 文件 agent 與第一個實作 agent 同時啟動。它讀取 `.kaizen/spec.md` 並為功能撰寫一個 VitePress 頁面，更新側邊欄設定，完成後提交。

**寫入了什麼：**
- `docs/guide/{feature}.md`
- 更新後的 `docs/.vitepress/config.ts`

**原因：** 事後撰寫的文件總是被推遲。並行執行意味著文件在驗收前就已完成。

---

## 重新啟動後繼續

因為所有狀態都存於 `.kaizen/` 檔案（git 追蹤），技能可以從任何階段繼續：

| `.kaizen/` 的狀態 | 從此繼續 |
|---|---|
| 無 `.kaizen/` 目錄 | 第 1 階段 |
| `spec.md` 存在，無 `test-strategy.md` | 第 2 階段 |
| `test-strategy.md` 存在，測試未全部通過 | 第 3 階段 |
| 所有測試通過，無驗收日誌條目 | 第 4 階段 |
| 驗收已記錄，文件缺失 | 第 5 階段 |

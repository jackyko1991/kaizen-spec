# 哲學理念

kaizen-spec 以**豐田生產系統（TPS）**為基礎，透過 Mary 與 Tom Poppendieck 在《精實軟體開發》（*Lean Software Development*）中的詮釋，將豐田概念直接對應至 kaizen-spec 所強制執行的軟體實踐。

---

## 為何選擇 TPS？

豐田耗費數十年，從實體生產線中消除浪費、缺陷與過度負荷。同樣的問題也出現在軟體開發中：程式碼在交付前不斷累積、缺陷向下游擴散、agents 負荷過重、脈絡在工作階段之間流失。TPS 提供了經過驗證的詞彙與解方。

核心洞見：**軟體中的浪費就是未交付的程式碼**。每一行存在但尚未交付價值的程式碼都是庫存——它耗費維護成本、承擔過時風險，並為每一個未來讀取它的 agent 增加脈絡負擔。

| 豐田 / TPS | JP | 軟體對應實踐 | 缺少時會發生什麼 |
|---|---|---|---|
| Muda — 浪費消除 | 無駄 | 未交付的程式碼就是庫存浪費 | 程式碼在觸達使用者前不斷累積維護成本與過時風險 |
| 即時生產（JIT） | ジャスト・イン・タイム | CI/CD — 拉式交付 | 大批次發布累積風險；缺陷在被發現前持續複合 |
| 自働化 — Jidoka | 自働化 | TDD — 測試拉下安燈繩 | 缺陷流入生產環境；無感測器可停止生產線 |
| 防呆 — Poka-Yoke | 防呆 | 靜態型別、linting、schema 驗證 | 錯誤在執行時或由使用者發現，而非在撰寫時攔截 |
| 改善 — 持續改善 | 改善 | 規格改善 — 測試失敗回饋至規格 | 規格與現實偏離；agents 反覆解決錯誤的問題 |
| 一個流 | 一個流 | 原子規格 — 一個 agent、一項任務、一個責任 | 大型脈絡視窗降低 agent 精確度；長任務無法並行 |
| 延遲決策 | — | 精實規格 — 即時設計 | 大量預先規格在實作前即已過時；過度工程被固化 |
| 標準作業 | 標準作業 | 狀態存於 `.kaizen/` 檔案，而非 agent 記憶 | 全新脈絡的 agents 無法接續；使用者必須從頭重新說明 |

## 五大原則的實踐

### 1. 規格先於程式碼（延遲決策）

立即開始撰寫程式碼的 agents 有一半的時間在解決錯誤的問題。花 5 分鐘對齊規格能節省數小時的返工。規格門閘是精實思維的「在最後責任時機做決策」——只有在擁有足夠資訊時，才做出正確決策的承諾。

### 2. 測試先紅後綠（自働化）

從未失敗過的測試，從未證明它能捕捉缺陷。自働化的意義是：機器偵測異常並停止——在 TDD 中，失敗的測試就是那個感測器。如果感測器從未被測試過，就不值得信任。

### 3. WIP 限制（一個流 + 平準化）

並行不是免費的。同時對一個功能跑 20 個 agents 會造成資源爭奪、合併衝突與輸出品質下降。WIP 限制強制執行一個流：少量任務完整通過系統後，才開始下一批。

### 4. 工作可視化（安燈 + 標準作業）

不在看板上的工作就不存在。每一項任務——無論多瑣碎——都要有一張看板卡片。如果 agent 被阻塞，它就拉下安燈繩（設定 `data-blocked="true"`）並停止，而非繼續產出有缺陷的輸出。

### 5. 狀態存於檔案，而非記憶（標準作業）

Agents 會重新啟動。工作階段會結束。記憶是短暫的。`.kaizen/` 目錄就是標準作業——任何全新脈絡的 agent 都能讀取這個單一事實來源，無需告知便能了解專案的確切進度。

## 精實開發 - 七大原則

精實開發（Lean Development）源自於豐田生產系統的「精益生產」理念。其核心思想是**消除浪費**——讓開發過程中的每一分鐘和每一行程式碼都能為客戶創造價值。

精實開發由 Mary 與 Tom Poppendieck 於 2003 年的著作《精實軟體開發》（*Lean Software Development*）中，將精益生產的理念引入軟體工程，並提出七大核心原則：

### 1. 消除浪費

去除不必要的功能、減少等待時間、避免過度處理（如冗余測試、過度文檔）。凡是不直接為客戶服務的，皆為 Muda（浪費）。

### 2. 內建品質

自動化測試與持續整合確保問題在發生當下便被修復，而非事後補救。品質不是末端的檢驗，而是每個步驟都被建立進去（Jidoka）。

### 3. 創造知識

鼓勵團隊學習、記錄實驗結果並分享經驗。`.kaizen/` 記錄檔與規格文件即是一種制度化的知識創造形式。

### 4. 延遲承諾

在擁有足夠資訊之前，不做不可逆的決策。保留彈性——規格門閘在此強制執行：範疇明確前，不開始實作。

### 5. 快速交付

縮短反饋迴路，盡快將產品交到使用者手中以驗證想法。週期時間越長，等待學習「是否建造了正確的東西」的時間就越長。

### 6. 尊重人員

賦予一線開發者決策權，信任其專業判斷。在 kaizen-spec 中：agents 被給予明確的脈絡和其任務範疇內的自主權；協調者不微管理實作細節。

### 7. 整體最佳化

關注整個價值流的流動，而非單一部門或步驟。一個快速實作但產生缺陷、拖慢審查的團隊，並非最佳化系統——它只是移動了瓶頸。

## 精實 vs. 敏捷

精實（Lean）與敏捷（Agile）常被相提並論，兩者都強調靈活性和快速迭代——但它們的著眼點不同：

| 特性 | 敏捷 | 精實 |
|---|---|---|
| 核心目標 | 透過迭代回應需求變化 | 透過消除浪費和優化流程提升效率 |
| 方法 | 小步驟、持續交付、客戶溝通 | 聚焦價值流流動；找出流程瓶頸 |
| 管理方式 | Scrum：固定週期、明確角色 | 看板：視覺化流程，限制 WIP |
| 對人員的看法 | 跨職能、自組織團隊 | 系統性流程改善與持續改善 |
| 對失敗的態度 | 快速失敗——透過頻繁失敗學習 | 降低錯誤與等待以降低成本 |

**兩者並不互斥。** 現代高效能團隊通常採用精實-敏捷混合模式：

- 用**敏捷思維**面對不確定的市場需求與頻繁的客戶互動。
- 用**精實工具**（價值流圖、看板）審視開發過程中是否有過多 WIP 或不必要的行政負擔。

簡言之：敏捷告訴你「如何有效地做對的事」，精實告訴你「如何有效率地把事做對」。

對於需要高穩定性與資源最佳化的專案——如基礎架構工作或受監管的系統（醫療影像、IEC 62443、DO-178C）——精實原則在降低無效投入方面具有最大的優勢。

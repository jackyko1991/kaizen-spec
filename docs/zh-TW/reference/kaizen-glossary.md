# 改善與看板術語
# 改善・看板 — 用語集

kaizen-spec 建立於豐田生產系統（Toyota Production System）的哲學之上。本頁以繁體中文說明整個技能與文件中使用的術語。

---

## 看板（Kanban）— 流動的科學

看板（看板，「信號牌」）是一種管理工作流程的排程系統。它不只是板上的卡片，更是使工作可視化並限制過載的紀律。

### WIP 限制 — 在製品限制（Work In Progress Limit）

**看板的靈魂。** 限制同時進行的任務數量。防止系統自我過載。當某一欄已滿時，新工作無法進入——必須先完成現有工作。

> 「停止開始，開始完成。」

在 kaizen-spec 中：`.kaizen/tasks.json` 中的 `wip_limits` 欄位限制並行 agent 數量。超過限制時，看板欄位會變紅（安燈訊號）。

---

### 交付周期時間（Lead Time）

從任務**建立**到**交付**所需的總時間。

```
交付周期時間 = completed_at − created_at
```

包含：在待辦清單等待的時間 + 主動工作時間。過長的交付周期時間意味著任務在被 agent 接手前長時間閒置。在看板卡片上以 **LT** 顯示。

---

### 週期時間（Cycle Time）

從工作**開始**到**完成**所需的時間。

```
週期時間 = completed_at − started_at
```

衡量執行效率。`交付周期時間 − 週期時間` = 排隊等待時間（等待的浪費）。在看板卡片上以 **CT** 顯示。

---

### 拉式系統（Pull System）— 引き取り / 拉式生産

工作由 agent 在有產能時**拉取**，而非不論負載如何強行推送。agent 只有在完成當前任務且有 WIP 空位時，才從待辦清單取下一項任務。

---

### 節拍時間（Takt Time）— タクト時間

滿足需求所需的工作交付速率。

```
節拍時間 = 可用時間 ÷ 使用者請求數量
```

使用 kaizen.log 的交付周期時間資料，檢查 agent 的並行程度是否符合節拍時間。

---

### 平準化（Heijunka）— 平準化（負載平衡）

平滑不均勻的工作量（不均）。kaizen-spec 透過 WIP 限制 + 拉式系統自動達成：任務以 `wip_limits` 個為單位穩定處理，而非爆發式地進行。

---

### 瓶頸（Bottleneck）— ボトルネック

最慢的階段，限制了整體的吞吐量。若審查欄始終滿載而待辦清單不斷增長，審查就是瓶頸——應先在那裡增加產能。

---

## 改善（Kaizen）— 消除浪費

改善（改善，「改變向好」）是持續、漸進的改進，不是一次性活動，而是永久的運作模式。

---

### 浪費（Muda）— 無駄

任何消耗資源而不產生價值的活動。七種類型，對應至 agentic 開發：

| 豐田浪費 | Agentic 等效物 | kaizen-spec 的應對 |
|---|---|---|
| 過剩生產 | 超出規格的程式碼 | 第一階段規格門控 |
| 等待 | Agent 等待輸入而閒置 | 安燈機制——快速浮現阻礙 |
| 缺陷 | 測試失敗或不符合規格的程式碼 | TDD——自働化偵測缺陷 |
| 動作 | Agent 重新讀取已處理的上下文 | 狀態儲存於 `.kaizen/` 檔案 |
| 運輸 | 過多 agent 相互傳遞輸出 | Worktree 隔離 |
| 庫存 | 待辦清單中堆積的任務 | WIP 限制 |
| 過度處理 | 為瑣碎的重新命名執行完整五階段流程 | SKIP 保護機制 |

第四階段驗收後的 **5S 清潔**會移除程式碼中的浪費：死碼、多餘注解、未使用的匯入。程式碼越少 = 未來 agent 需要載入的上下文越少 = token 成本越低。

---

### 不均（Mura）— 斑（不均勻）

不規則的流動——某天有一百個任務，下一天卻什麼都沒有。平準化（WIP 限制 + 拉式系統）是解藥。

---

### 過負荷（Muri）— 無理（過度負擔）

強迫 agent 超出產能。在資源只支援 3 個的情況下並行啟動 20 個 agent 會造成系統抖動和輸出品質下降。WIP 限制直接防止過負荷。

---

## 組合與進階術語

### 安燈（Andon）— 安燈（信號燈）

視覺停止訊號。在豐田工廠中，任何工人在發現問題時都可以拉安燈繩，停止生產線。

在 kaizen-spec 中：被阻擋的 agent 在看板卡片上設置 `data-blocked="true"`，向 `kaizen.log` 記錄 `WARN`，並**停止**。看板顯示紅色 **安燈 ⚠ BLOCKED** 徽章。測試套件在測試失敗時也會自動拉起安燈繩（自働化）。

---

### 自働化（Jidoka）— 自働化（自律自動化）

內建缺陷智能的自動化。機器偵測異常並停止，而不是繼續生產有缺陷的輸出。

**在 kaizen-spec 中，自働化 = TDD 循環：**

| TDD 階段 | 自働化角色 |
|---|---|
| 撰寫失敗測試（紅） | 定義「缺陷」的含義 |
| 執行測試套件 | 機器自主偵測缺陷 |
| 測試失敗 → CI 停止 | 自動拉起安燈 |
| 實作（綠） | 在機器監督下修復 |
| 重構 → 重新測試 | 確認無回歸 |

從未失敗過的測試，從未證明過它能偵測到缺陷——不能作為可信的自働化感測器。

---

### 防呆（Poka-Yoke）— ポカヨケ（防錯）

設計系統使錯誤**不可能**發生，而非僅是不太可能。

在 kaizen-spec 中：`tasks.json` 有必要的結構；階段門控防止在規格提交前寫程式碼；WIP 強制執行是結構性的（非建議性）。這些限制就是防呆——agent 無法意外繞過它們。

在軟體中：靜態型別、結構驗證和 linting 就是防呆。

---

### 現地現物（Genchi Genbutsu）— 現地現物（親臨現場）

「前往實際地點，看見實際事物。」診斷問題時絕不依賴摘要。

**技能規則：** 當任務受阻或測試失敗時，agent 必須直接讀取原始堆疊追蹤/日誌行——不向其他 agent 要求摘要。**現場**（現場）是實際的錯誤輸出，而非對它的描述。

---

### 現場（Gemba）— 現場（真實之地）

價值實際創造的地方。對於軟體：運行環境、實際測試套件輸出、實際 `.kaizen/` 狀態檔案。不是 agent 對事件的記憶。

---

### 標準作業（Standard Work）— 標準作業

關於如何完成工作的文件化、已達成共識的基準。若流程未標準化，改善就無從改進。`.kaizen/` 目錄結構就是 kaizen-spec 的標準作業。

---

### PDCA → TDD — 計画・実行・評価・改善

戴明改善循環直接對應至 TDD：

| PDCA | TDD |
|---|---|
| 計劃 | 撰寫失敗測試——定義成功 |
| 執行 | 實作程式碼 |
| 查核 | 執行測試套件 |
| 行動 | 若綠色則重構；若紅色則修復並重複 |

---

### 5S — 職場環境整備

職場組織紀律。應用於程式碼和 agent 上下文：

| 5S 步驟 | 日文 | 程式碼 / agent 含義 |
|---|---|---|
| 整理 | Seiri | 移除死碼、未使用的匯入、過時注解 |
| 整頓 | Seiton | 一致的命名、合理的檔案組織 |
| 清掃 | Seiso | 移除暫存檔、測試產物、`.kaizen/*.tmp` |
| 清潔 | Seiketsu | Linting、型別檢查、結構驗證 |
| 躾 | Shitsuke | CI/CD 在每次合併時強制執行 5S |

kaizen-spec 在第四階段驗收後執行 5S 清潔 agent。目標：從程式碼庫中移除浪費，讓未來的 agent 工作階段載入更少上下文，花費更少 token。

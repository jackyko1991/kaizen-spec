# 5つのフェーズ

kaizen-spec は5つのフェーズを順番に強制します。フェーズはスキップできません。各フェーズが次のフェーズのゲートになります。

```mermaid
flowchart LR
  S1["📋 フェーズ1\n仕様整合"]
  S2["🔴 フェーズ2\nテスト赤"]
  S3["⚙️ フェーズ3\n実装"]
  S4["✅ フェーズ4\n受け入れ"]
  S5["📖 フェーズ5\nドキュメント"]

  S1 -->|"spec.md\nコミット済"| S2
  S2 -->|"全テスト\n失敗"| S3
  S3 -->|"全テスト\n通過"| S4
  S4 -->|"受け入れ\n記録済"| S5

  style S1 fill:#1f2d45,stroke:#388bfd,color:#cdd9e5
  style S2 fill:#3d1f1f,stroke:#f85149,color:#cdd9e5
  style S3 fill:#1f2d1f,stroke:#56d364,color:#cdd9e5
  style S4 fill:#1f2d1f,stroke:#56d364,color:#cdd9e5
  style S5 fill:#2d2520,stroke:#e3b341,color:#cdd9e5
```

---

## フェーズ1 — 仕様整合

**何が起きるか：** スキルが一連の構造化された質問を1つずつ行います。何を作りたいか、出力はどうあるべきか、スコープに含まれるもの、明示的に除外されるもの、どんなリスクがあるか。

**書き込まれるもの：** `.kaizen/spec.md` — コミットされた合意済みの仕様。

**ゲート：** `.kaizen/spec.md` が git にコミットされるまでフェーズ2は開始されません。

**なぜか：** 仕様なしにコードを書くエージェントは誤った問題を解きます。このフェーズは2〜5分かかり、何時間もの手戻りを防ぎます。

---

## フェーズ2 — テスト戦略

**何が起きるか：** スキルがプロジェクトのスタック（TypeScript・Python・Go など）を検出し、テストフレームワークを推薦します。あなたが確認するか代替を選択します。テスト作成エージェントが仕様のすべての受入基準に対してテストを書き、それらがすべて**失敗する**ことを確認してから次に進みます。

**書き込まれるもの：**
- `.kaizen/test-strategy.md` — フレームワーク・インストールコマンド・テストファイルパス・すべてのテストのリスト
- `tests/` — 実際のテストファイル（すべて失敗）

**ゲート：** この段階でテストが1つでも通過していれば、スキルは停止して調査を求めます。実装前にテストが通ることは、テストが誤っているか、機能がすでに存在することを意味します。

**なぜか：** コードを書いた後に書かれたテストは、合意した仕様ではなく書かれたコードをテストする傾向があります。先にテストを書くことで、正直なカバレッジが強制されます。

---

## フェーズ3 — 実装

**何が起きるか：** スキルが仕様を独立したタスクに分解し、`.kaizen/tasks.json` に書き込み、並列にサブエージェントを生成します。1タスクにつき1エージェント、各エージェントは git ワークツリー内で作業します。エージェントはテストが通るまで実装を続け、完了したら自分自身を完了とマークします。

**書き込まれるもの：**
- `.kaizen/tasks.json` — ステータス・エージェントのアサイン・タイムスタンプを含むタスクリスト
- `.kaizen/board.html` — ライブかんばんボード（ブラウザで開いて監視できます）
- `.kaizen/kaizen.log` — 状態遷移ごとに1つの syslog エントリ
- 実装コード（git ワークツリー内で作成し、完了時にマージ）

**WIP制限：** 同時に「実行中」にできるタスクは最大3件、「レビュー」は最大2件です。上限を超えるとカラムのヘッダーが赤くなり、移動がブロックされます。

**アンドン・コード：** エージェントが詰まった場合、ボードカードにブロックフラグをセットし、`kaizen.log` に `WARN` エントリをログします。あなたに通知され、ブロックを解除できます。

**ゲート：** フェーズ4が始まる前にすべてのテストが通過しなければなりません。

---

## フェーズ4 — 受入

**何が起きるか：** スキルがテストスイート全体を実行します。何か失敗した場合、完了とは宣言せず、フェーズ3に戻って修正します。手動検証が必要な場合（例：UIの目視確認）、特定の項目について確認を求めます。

**書き込まれるもの：**
- `.kaizen/kaizen.log` — 受入結果と総サイクルタイム

**ゲート：** すべてのテストが緑 + 必要な手動確認がすべて完了。

**なぜか：** 「完了」とは動作するソフトウェアを意味し、「自分のマシンでは動く」ではありません。

---

## フェーズ5 — ドキュメント（フェーズ3と並行）

**何が起きるか：** ドキュメントエージェントが最初の実装エージェントと同時に起動します。`.kaizen/spec.md` を読み、機能のための VitePress ページを書きます。サイドバーの設定を更新し、完了したらコミットします。

**書き込まれるもの：**
- `docs/guide/{feature}.md`
- 更新された `docs/.vitepress/config.ts`

**なぜか：** 後から書かれたドキュメントは常に先延ばしにされます。並行して実行することで、受入前に完成します。

---

## 再起動後の再開

すべての状態は `.kaizen/` ファイル（git管理）に保存されているため、スキルはどのフェーズからでも再開できます：

| `.kaizen/` の状態 | 再開フェーズ |
|---|---|
| `.kaizen/` ディレクトリなし | フェーズ1 |
| `spec.md` が存在し、`test-strategy.md` がない | フェーズ2 |
| `test-strategy.md` が存在し、テストが全部通っていない | フェーズ3 |
| テストがすべて通っており、受入ログエントリがない | フェーズ4 |
| 受入がログされており、ドキュメントが未完成 | フェーズ5 |

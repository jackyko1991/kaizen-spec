# 哲学

kaizen-spec は、メアリー・ポッペンディークとトム・ポッペンディークが著書 *Lean Software Development* でソフトウェア開発に翻訳した**トヨタ生産方式（TPS）**に基づいています。トヨタの各概念は、このスキルが強制するソフトウェアプラクティスに直接対応しています。

---

## なぜ TPS なのか？

トヨタは数十年をかけて、物理的な生産ラインからムダ・欠陥・過負荷を排除してきました。ソフトウェアにも同じ問題が存在します。コードはリリース前に蓄積し、欠陥は下流で複合し、エージェントは過負荷になり、セッション間でコンテキストが失われます。TPS は実証済みの語彙と実証済みの解決策を提供します。

核心的な洞察：**ソフトウェアにおけるムダとは、未出荷のコードである**。存在するが価値を届けていないコードはすべて在庫です。保守コストがかかり、陳腐化リスクを抱え、将来それを読むすべてのエージェントにコンテキスト負荷を加えます。

## TPS → ソフトウェア対応表

| トヨタ / TPS | JP | ソフトウェアの対応概念 | それなしで何が壊れるか |
|---|---|---|---|
| ムダ — 無駄の排除 | 無駄 | 未出荷コードは在庫ムダ | コードはユーザーに届く前から保守コストと陳腐化リスクが積み上がる |
| ジャスト・イン・タイム（JIT） | ジャスト・イン・タイム | CI/CD — プル型デリバリー | 大バッチリリースはリスクを蓄積し、欠陥が検出前に複合する |
| 自働化 — ジドウカ | 自働化 | TDD — テストがアンドン・コードを引く | 欠陥が本番まで流れる。ラインを止めるセンサーがない |
| ポカヨケ — 防呆 | 防呆 | 静的型付け・Lint・スキーマ検証 | エラーが書いた時点ではなく実行時やユーザーによって発見される |
| 改善 — 継続的改善 | 改善 | 仕様改善 — テストの失敗が仕様にフィードバックされる | 仕様が現実からずれ、エージェントが繰り返し誤った問題を解く |
| 一個流れ | 一個流 | アトミック仕様 — 1エージェント・1タスク・1責務 | 大きなコンテキストウィンドウはエージェントの精度を下げ、長いタスクは並列化できない |
| 決定を遅らせる | — | リーン仕様 — ジャスト・イン・タイム設計 | 大規模前置仕様は実装前に陳腐化し、過剰設計が組み込まれる |
| 標準作業 | 標準作業 | 状態は `.kaizen/` ファイルに保存し、エージェントの記憶には頼らない | 新規コンテキストのエージェントが再開できず、ユーザーがゼロからコンテキストを説明し直す必要がある |

## 実践における5つの原則

### 1. コードより先に仕様を（決定を遅らせる）

すぐにコーディングを始めるエージェントは、半分の確率で誤った問題を解きます。5分の仕様整合により、何時間もの手戻りを防げます。仕様ゲートはリーンの「最後の責任ある瞬間に決定する」原則です。正しい決定を下せるだけの情報が揃ったときに初めて、解決策にコミットします。

### 2. テストは赤から始める（自働化）

一度も赤にならなかったテストは、欠陥を捕捉できることを証明したことがありません。自働化とは、機械が異常を検知して停止することを意味します。TDD では、失敗するテストがそのセンサーです。センサーがテストされていなければ、信頼できません。

### 3. 仕掛品の上限（一個流れ + 平準化）

並列処理はタダではありません。1つの機能に20のエージェントを同時に走らせると、スラッシングやマージコンフリクト、出力品質の低下が起きます。WIP制限は一個流れを強制します。次のバッチを始める前に、少数のタスクをシステム全体を完全に通過させます。

### 4. 仕事を見える化する（アンドン + 標準作業）

ボードに載っていない仕事は存在しません。些細なタスクも複雑なタスクも、すべてかんばんカードになります。エージェントがブロックされたら、アンドン・コードを引き（`data-blocked="true"` をセット）、欠陥品を生産し続けるのではなく停止します。

### 5. 状態はファイルに、記憶には頼らない（標準作業）

エージェントは再起動します。セッションは終わります。記憶は揮発性です。`.kaizen/` ディレクトリが標準作業であり、新規コンテキストのエージェントが、何も教えられなくてもプロジェクトの現状を正確に把握できる唯一の真実のソースです。

## リーン開発 - 7つの原則

リーン開発はトヨタ生産方式の「リーン生産」理念に由来します。その核心思想は**ムダの排除**です。開発プロセスにおけるすべての時間とコードが顧客に価値をもたらすようにします。

リーン開発は、メアリー・ポッペンディークとトム・ポッペンディークが 2003 年の著書 *Lean Software Development* でリーン生産の考え方をソフトウェア工学に翻訳し、7つのコア原則を提示したことから始まります：

### 1. ムダの排除

不要な機能を取り除き、待ち時間を減らし、過剰処理（冗長なテスト、過剰なドキュメント）を避ける。顧客に直接価値をもたらさないものはすべてムダです。

### 2. 品質を作り込む

自動テストと継続的インテグレーションにより、問題は発生した瞬間に修正されます。品質は最後に検査して確認するものではなく、すべてのステップで作り込まれます（自働化）。

### 3. 知識を創造する

チームの学習を促進し、実験結果を記録し、経験を共有する。`.kaizen/` ログと仕様ファイルは、制度化された知識創造の一形態です。

### 4. 決定を遅らせる

十分な情報が得られるまで、取り返しのつかない決定をしない。柔軟性を維持する。仕様ゲートはこれを強制します：スコープが明確になるまで実装を始めない。

### 5. 速く届ける

フィードバックサイクルを短縮する。アイデアを検証するため、製品を素早くユーザーに届ける。サイクルタイムが長いほど、「正しいものを作ったか」を学ぶまでの待機時間も長くなります。

### 6. 人を尊重する

現場の開発者に意思決定権を与え、専門知識を信頼する。kaizen-spec では、エージェントには明確なコンテキストとタスク範囲内の自律性が与えられます。オーケストレーターは実装の細部をマイクロマネジメントしません。

### 7. 全体を最適化する

単一の部門やステップだけでなく、バリューチェーン全体のフローに注目する。欠陥を生み出してレビューを遅らせる高速な実装チームは最適化されたシステムではありません。ボトルネックを移動させただけです。

## リーン vs. アジャイル

リーンとアジャイルはしばしば並べて語られ、どちらも柔軟性と迅速なイテレーションを重視しています。しかし、その焦点は異なります：

| 特性 | アジャイル | リーン |
|---|---|---|
| 核心目標 | イテレーションを通じて変化する要件に対応する | ムダの排除とフローの最適化により効率を高める |
| アプローチ | 小さなステップ・継続的デリバリー・顧客とのコミュニケーション | バリューストリームのフローに注目し、プロセスのボトルネックを見つける |
| 管理スタイル | スクラム：固定スプリント・明確な役割 | かんばん：プロセスフローを可視化し、WIP を制限する |
| 人への視点 | クロスファンクショナルな自己組織化チーム | 系統的なプロセス改善と継続的改善 |
| 失敗への姿勢 | 速く失敗 - 頻繁な失敗から学ぶ | エラーと待機を減らしてコストを下げる |

**両者は相互排他的ではありません。** 高パフォーマンスな現代チームは通常、リーン・アジャイルのハイブリッドを採用します：

- **アジャイルのマインドセット**で、不確かな市場要件と頻繁な顧客との対話に対処する。
- **リーンツール**（バリューストリームマップ、かんばん）を使って、開発プロセスに過剰な WIP や不要な管理上のオーバーヘッドがないかを検討する。

要するに、アジャイルは「正しいことを効果的に行う方法」を教え、リーンは「物事を効率的に正しく行う方法」を教えます。

高い安定性とリソース最適化が求められるプロジェクト - インフラ作業や規制システム（医療画像、IEC 62443、DO-178C）など - では、リーン原則が無駄な努力を削減する上で最大の利点をもたらします。

# 改善・看板 — 用語集

kaizen-specはトヨタ生産方式（TPS）の哲学に基づいています。このページでは、スキルとドキュメント全体で使用される用語を日本語で説明します。

---

## かんばん（看板）— フローの科学

かんばん（看板、「信号板」）は、作業フローを管理するためのスケジューリングシステムです。単なるボード上のカードではなく、作業を可視化し過負荷を制限するための規律です。

### WIP制限 — 仕掛品制限（Work In Progress Limit）

**かんばんの魂。** 同時にアクティブにできるタスクの数を制限します。システムの過負荷を防ぎます。列が満杯になると、新しい作業は入れません——既存の作業を先に完了させる必要があります。

> 「着手をやめて、完了を始めよ。」

kaizen-specでは：`.kaizen/tasks.json`の`wip_limits`フィールドが並行エージェント数を制限します。制限を超えると、ボードの列が赤くなります（安燈シグナル）。

---

### リードタイム（Lead Time）— 交付周期時間

タスクが**作成**されてから**納品**されるまでの総時間。

```
リードタイム = completed_at − created_at
```

バックログで待機している時間 + アクティブな作業時間を含みます。リードタイムが長い場合、エージェントが引き取る前にタスクが放置されています。ボードカードに **LT** として表示されます。

---

### サイクルタイム（Cycle Time）— 週期時間

作業が**開始**されてから**完了**されるまでの時間。

```
サイクルタイム = completed_at − started_at
```

実行効率を測定します。`リードタイム − サイクルタイム` = 待ち行列待機時間（待機のムダ）。ボードカードに **CT** として表示されます。

---

### 引き取り方式（Pull System）— 引き取り / 拉式生産

作業はエージェントが余力を持っている時に**引き取られ**——負荷に関わらず押し付けられることはありません。エージェントは現在のタスクが完了してWIPスロットが空いた時にのみ、バックログから次のタスクを引き取ります。

---

### タクトタイム（Takt Time）— タクト時間

需要を満たすために作業を届けなければならない速度。

```
タクトタイム = 利用可能時間 ÷ ユーザーリクエスト数
```

kaizen.logのリードタイムデータを使用して、エージェントの並行性がタクトタイムを満たしているか確認します。

---

### 平準化（Heijunka）— 平準化（負荷平準化）

不均一な作業量（ムラ）を平滑化すること。kaizen-specはWIP制限＋引き取り方式によって自動的にこれを実現します：タスクは突発的にではなく、`wip_limits`個ずつ着実に処理されます。

---

### ボトルネック（Bottleneck）— ボトルネック

総スループットを制限する最も遅い段階。レビュー列が常に満杯でバックログが増え続けているなら、レビューがボトルネックです——他の場所より先にそこの能力を増やしましょう。

---

## 改善（Kaizen）— ムダの排除

改善（改善、「より良い変化」）は継続的で漸進的な改善です。一度きりのイベントではなく、永続的な運営モードです。

---

### ムダ（Muda）— 無駄（廃棄物）

価値を生み出さずにリソースを消費する活動。7つの種類が、エージェント開発に対応します：

| トヨタのムダ | エージェント相当物 | kaizen-specの対策 |
|---|---|---|
| 過剰生産 | 仕様にないコード | フェーズ1の仕様ゲート |
| 待機 | 入力待ちでアイドルのエージェント | 安燈コード——障害を素早く表面化 |
| 欠陥 | テスト失敗または仕様不一致のコード | TDD——自働化が欠陥を検出 |
| 動作 | 以前処理したコンテキストを再読み込みするエージェント | `.kaizen/`ファイルへの状態保存 |
| 運搬 | 多すぎるエージェント間での出力受け渡し | ワークツリー分離 |
| 在庫 | バックログに積み重なるタスク | WIP制限 |
| 過剰処理 | 些細なリネームに対する完全な5フェーズ手順 | SKIPガード |

フェーズ4の **5Sクリーンアップ**はコードからムダを取り除きます：デッドコード、冗長なコメント、未使用のインポート。コードが少ない = 将来のエージェントが読み込むコンテキストが少ない = トークンコストが低い。

---

### ムラ（Mura）— 斑（不均一性）

不規則なフロー——ある日は100タスク、次の日は何もない。平準化（WIP制限＋引き取り方式）が解決策です。

---

### ムリ（Muri）— 無理（過負荷）

エージェントを能力以上に押し付けること。3つしかサポートできないリソースで20の並行エージェントを起動するとスラッシングと出力品質低下を引き起こします。WIP制限がムリを直接防ぎます。

---

## 組み合わせた・高度な用語

### 安燈（Andon）— 安燈（信号灯）

視覚的な停止シグナル。トヨタの工場では、どの作業者も問題を検出した時に安燈コードを引いて生産ラインを止めることができます。

kaizen-specでは：ブロックされたエージェントはボードカードに`data-blocked="true"`を設定し、`kaizen.log`に`WARN`を記録し、**停止**します。ボードには赤い**安燈 ⚠ BLOCKED**バッジが表示されます。テストスイートがテスト失敗時に安燈コードを自動的に引くこともあります（自働化）。

---

### 自働化（Jidoka）— 自働化（自律的自動化）

欠陥検知インテリジェンスを組み込んだ自動化。機械は異常を検出して停止し、欠陥品の生産を続けることはありません。

**kaizen-specでは、自働化 = TDDサイクル：**

| TDDフェーズ | 自働化の役割 |
|---|---|
| 失敗するテストを書く（レッド） | 「欠陥」の定義 |
| テストスイートを実行する | 機械が自律的に欠陥を検出 |
| テスト失敗 → CIが停止 | 安燈が自動的に引かれる |
| 実装（グリーン） | 機械の監督下で修正 |
| リファクタリング → 再テスト | 回帰がないことを確認 |

一度も失敗したことのないテストは、欠陥を検出できることを証明したことがない——自働化センサーとして信頼することはできません。

---

### ポカヨケ（Poka-Yoke）— ポカヨケ（誤り防止）

エラーが**不可能**になるようにシステムを設計すること（単に起こりにくいだけでなく）。

kaizen-specでは：`tasks.json`には必須スキーマがあり、フェーズゲートは仕様コミット前のコーディングを防ぎ、WIP強制は構造的です（助言的ではありません）。これらの制約はポカヨケです——エージェントはうっかり回避することができません。

ソフトウェアでは：静的型付け、スキーマバリデーション、リンティングがポカヨケです。

---

### 現地現物（Genchi Genbutsu）— 現地現物（行って見る）

「実際の場所に行き、実際のものを見る。」問題を診断する時は要約に頼らないこと。

**スキルルール：** タスクがブロックされているかテストが失敗している場合、エージェントは生のスタックトレース/ログラインを直接読む必要があります——他のエージェントに要約を求めないこと。**現場**（現場）は実際のエラー出力であり、その説明ではありません。

---

### 現場（Gemba）— 現場（実際の場所）

価値が実際に生み出される場所。ソフトウェアでは：実行環境、実際のテストスイート出力、実際の`.kaizen/`状態ファイル。エージェントのイベント記憶ではありません。

---

### 標準作業（Standard Work）— 標準作業

作業の進め方に関する文書化された、合意された基準。改善は標準化されていないプロセスを改善できません。`.kaizen/`ディレクトリ構造がkaizen-specの標準作業です。

---

### PDCA → TDD — 計画・実行・評価・改善

デミング改善サイクルはTDDに直接対応します：

| PDCA | TDD |
|---|---|
| 計画 | 失敗するテストを書く——成功を定義する |
| 実行 | コードを実装する |
| 評価 | テストスイートを実行する |
| 改善 | グリーンならリファクタリング；レッドなら修正して繰り返す |

---

### 5S — 職場環境整備

職場組織の規律。コードとエージェントのコンテキストに適用：

| 5Sステップ | 日本語 | コード / エージェントの意味 |
|---|---|---|
| 整理 | Seiri | デッドコード、未使用のインポート、古いコメントを削除 |
| 整頓 | Seiton | 一貫した命名、論理的なファイル構成 |
| 清掃 | Seiso | 一時ファイル、テストアーティファクト、`.kaizen/*.tmp`を削除 |
| 清潔 | Seiketsu | リンティング、型チェック、スキーマバリデーション |
| 躾 | Shitsuke | CI/CDがすべてのマージ時に5Sを強制 |

kaizen-specは受入後のフェーズ4で5Sクリーンアップエージェントを実行します。目標：コードベースからムダを取り除き、将来のエージェントセッションがより少ないコンテキストを読み込み、より少ないトークンコストで作業できるようにすること。

---
